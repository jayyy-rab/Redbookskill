# P0 Program Run and Output Stability Rules

## 一、运行稳定性原则
1. 每次运行必须使用固定入口命令，不允许临时拼接命令。
2. 每次运行必须读取同一个配置文件（如 `config.yaml` / `task_config.json`）。
3. 不允许在代码中写死临时路径、临时账号、临时商品名。
4. 所有输入参数必须显式传入：
- `product_image`
- `product_name` / `product_id`
- `keyword`
- `dry_run`
- `task_id`
- `output_dir`
5. 每次运行必须生成唯一 `task_id`。
6. 每次运行的日志、截图、报告必须保存到独立目录。
7. 不允许覆盖上一次运行结果。
8. 所有步骤必须支持重复执行。
9. 同一个 Step 重跑时，必须先检查已有中间产物。
10. 已成功的 Step 不允许重复执行，除非用户明确要求。
11. 任意 Step 失败后，后续 Step 必须 `blocked` / `skipped`，不允许继续执行。
12. 不允许把异常吞掉后继续跑。
13. 不允许无证据标记 `success`。
14. 不允许真实发布，除非用户明确关闭 `dry_run`。

## 二、环境稳定性规则
每次正式运行前，必须先做环境自检：
1. 检查 Python / Node 版本。
2. 检查依赖是否安装完整。
3. 检查 Chrome / Edge 是否可用。
4. 检查 CDP / Playwright / Selenium 是否可连接。
5. 检查登录状态是否有效。
6. 检查输入文件是否存在、可读、非空。
7. 检查输出目录是否可写。
8. 检查网络是否可访问目标页面。
9. 检查 `dry_run` 是否开启。
10. 检查发布保护是否生效。

任意环境检查失败，必须停止，不允许继续执行业务流程。

## 三、输出稳定性规则
所有报告必须使用固定结构，不允许自由发挥。

每个 Step 的输出必须包含：

```json
{
  "step": "Step X",
  "name": "步骤名称",
  "status": "success / failed / manual_review / skipped / blocked",
  "input": {},
  "output": {},
  "checks": [],
  "evidence": {
    "log_path": "",
    "screenshot_path": "",
    "file_path": ""
  },
  "error": {
    "code": "",
    "message": "",
    "root_cause": ""
  },
  "next_action": ""
}
```

## 四、成功判定规则
一个 Step 只有同时满足以下条件，才允许标记 `success`：
1. 执行动作完成。
2. 校验规则全部通过。
3. 有日志证据。
4. 页面步骤有截图或 DOM 证据。
5. 文件步骤有文件校验证据。
6. AI 生成内容有质量检测证据。
7. 没有 `unresolved_error`。
8. 没有 `manual_review_required`。

否则必须标记 `failed` / `manual_review` / `blocked`。

## 五、随机性控制规则
1. 文案生成必须固定输出格式。
2. 文案字数必须严格校验。
3. 图片处理参数必须写入配置文件。
4. 商品选择必须优先使用 `product_id`。
5. 没有 `product_id` 时，使用 `product_name` 完全匹配。
6. 不允许随机选择商品。
7. 不允许默认选择第一个结果。
8. 不允许 AI 自己猜测缺失信息。
9. 缺少关键输入时，必须 `manual_review`。
10. 所有 AI 生成结果必须保存原始输出和校验结果。

## 六、重试规则
1. 只允许对明确的临时性错误重试：
- 网络超时
- 页面加载超时
- 元素短暂不可见
- 下载超时
2. 不允许对业务错误盲目重试：
- 商品不存在
- 文案不合格
- 输入缺失
- 图片损坏
- 登录失效
3. 每个 Step 最多重试 1～3 次，必须写清楚。
4. 每次重试必须记录 `retry_count`。
5. 重试后仍失败，必须停止。
6. 不允许无限循环重试。

## 七、报告稳定性规则
每次运行结束后必须生成以下文件：
1. `run_report.json`：机器可读的结构化报告。
2. `run_report.md`：人类可读的运行报告。
3. `bug_list.md`：本轮发现的问题列表。
4. `evidence_index.json`：日志、截图、文件证据索引。
5. `state.json`：当前任务状态，支持断点续跑。

## 八、禁止行为
1. 禁止没有环境自检就开始跑。
2. 禁止没有状态文件就断点续跑。
3. 禁止修改成功 Step 的结果。
4. 禁止覆盖历史报告。
5. 禁止缺少证据还输出 `success`。
6. 禁止用 mock success 代替真实验证。
7. 禁止为了通过测试放宽校验。
8. 禁止失败后继续执行后续 Step。
9. 禁止输出格式每次变化。
10. 禁止真实发布。
