# 验证规则

## 核心原则

**单点验证 ≠ 邻接回归 ≠ 主流程通过 ≠ 客户级 dry-run ≠ 真实发布。**

这五个层级各自独立，低层级通过不代表高层级能通过。

---

## 层级定义

### 第 1 层：单点验证

只证明某个函数在当前页面状态下可用。

- 示例：`_query_node_id('input[type="file"]')` 返回 nodeId > 0
- 限制：只能说明该函数机械可调用，不代表它在主流程调用链中能正确执行
- 页面状态、前置步骤、调用顺序不同，结果可能完全不同

### 第 2 层：邻接回归

证明两个步骤的输出和输入能正确对接。

- 示例：Step9 的图片路径能被 Step10 的 `_upload_images` 正常接收和上传
- 检查点：上游输出的字段名、类型、格式是否与下游输入一致
- 常见失败：上游输出是相对路径，下游期望绝对路径

### 第 3 层：主流程 dry-run

证明真实调用链基本可用。

- 必须使用 `publish_pipeline.py` 或 `dryrun_step1_10_runner.py`
- 必须添加 `--preview` 保护，不允许点击最终发布
- 必须使用与客户级相同的参数、端口、账号、profile
- 不允许用临时脚本替代主流程

### 第 4 层：客户级 dry-run

证明接近交付。

- 使用客户提供的输入（标题、正文、图片、商品名）
- 通过全部验证后，才能考虑开启真实发布

### 第 5 层：真实发布

仅在客户级 dry-run 通过且用户明确授权后允许。

---

## 规则

### 规则 1：临时验证脚本不能替代主流程验证

如果验证脚本和主流程存在以下任意差异，验证结论不能代表主流程：

- 使用不同的参数（`-depth=0` vs `{"depth": 1}`）
- 使用不同的 DOM 查询方式（JS `evaluate` vs CDP `DOM.querySelector`）
- 使用不同的点击方式（CDP 鼠标点击 vs JS `.click()`）
- 使用不同的端口、账号、profile
- 跳过了主流程必经的步骤（如 tab 点击、SPA 导航）

### 规则 2：层级不可跳过

不允许从单点验证直接跳到客户级 dry-run。必须逐级通过。

### 规则 3：失败必须降级诊断

主流程 dry-run 失败时，降级到单点验证缩小范围。确认单点通过后，再逐级恢复。

### 规则 4：页面状态必须在诊断中记录

每个验证步骤必须记录：
- 当前 URL
- 页面是否有遗留内容（已上传图片 / 商品卡片）
- CDP target 列表
- Chrome profile
- 端口

### 规则 5：截图和 DOM 不是可选项

每个关键页面节点必须截图。
涉及页面 DOM 查询的步骤必须保存 DOM dump。

---

## 违规后果

1. 用临时脚本通过代替主流程通过的，该验证无效。
2. 跳过层级验证直接跑客户级 dry-run 的，结果不认。
3. 验证失败后不降级诊断直接修代码的，该次修改无效。
